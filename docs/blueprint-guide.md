# Blueprint Guide

> Learn how to use Engineering Project Template blueprints effectively.

---

# Overview

Blueprints are reusable starting points for project documentation.

They are designed to be copied into a project and adapted to its specific needs.

Each blueprint represents a single responsibility.

Only use the blueprints that provide value for your project.

---

# Creating a New Project

1. Create the project repository.
2. Copy the Core blueprints.
3. Add additional blueprints only when they become necessary.
4. Replace placeholder values.
5. Remove instructional comments, if any.
6. Commit the initial project structure.

---

# Growing a Project

Projects should evolve progressively.

Do not create documents before they become useful.

Introduce new blueprints only when the project demonstrates a real need.

---

# Removing a Blueprint

A blueprint-derived document may be removed when:

* its responsibility no longer exists;
* another document legitimately replaces it;
* it no longer provides lasting value.

Never move information without preserving the project's single source of truth.

---

# Updating a Blueprint

Blueprints belong to the Engineering Project Template.

Projects may adapt their own documents after initialization.

Updating the template does not automatically require updating existing projects.

Projects remain responsible for their own documentation after creation.

---

# Best Practices

* Start with the smallest useful documentation set.
* Keep every document focused on one responsibility.
* Prefer references over duplicated information.
* Remove unused sections.
* Keep documentation concise.
* Let projects evolve naturally.

---

# Anti-Patterns

Avoid:

* creating documents "just in case";
* duplicating information across documents;
* modifying blueprints to fit a single project;
* treating every project as identical;
* copying every available blueprint into every project.

---

# Guiding Principle

Engineering Project Template provides building blocks, not obligations.

Projects should remain as simple as possible while staying as structured as necessary.
